# Security invariants (do not weaken)

- **Input contract — h3 decodes `event.url.pathname` exactly once** (`H3Event` constructor: `decodeURI`, with `%25` pre-escaped to `%2525` so a percent never unwraps). Every threat model here starts from what that leaves us, **not** from a raw wire path. By the time a pathname reaches this package: `%2e` → `.` and `%5c` → `\` → `/` (URL re-parse normalizes the backslash), with the revealed `.`/`..` **already resolved** — so a dot-segment traversal is gone before any rule matches; `%2f` stays opaque (`decodeURI` preserves reserved chars); `%252f`/`%252e` stay literal at any nesting depth, and no correctly single-decoding downstream will unwrap them either. **`%2f` is therefore the only encoded form that can still surface a traversal downstream** (`/api/orders/..%2fadmin` — dots decoded, separator not), and it is what dual-path matching and the scope checks exist for. Pinned by `test/h3-decode.test.ts`; if that suite fails, h3 changed its decode contract and every invariant below must be re-derived. Note the word "raw" below means _the once-decoded pathname h3 serves_ — the package never sees the wire-encoded path. `%2e`/`%5c`/`%25`-nesting handling in matching is **defense-in-depth for non-h3 callers** of the exported matchers, not an h3-reachable path.
- **Dual-path matching** (`src/match.ts`): every request is matched on the served pathname AND (if different) its canonical + slash-merged canonical forms. The canonical passes may **add** a rule the raw path missed and **override** one it resolved **only with an equal-or-more-specific pattern — never delete, never downgrade**; each path's `false` reset is resolved independently before the union. The override is specificity-guarded by `canOverride` (a `rou3` `compareRoutes` predicate injected by `createRouteRulesMatcher` into `createMatcherFromFind` → `mergeMatchedRouteRules`; kept out of `merge.ts` so compiled matchers stay `rou3`-free and tree-shakeable — a compiled matcher that does not wire it falls back to unconditional override). This is what blocks encoded-separator bypasses **and** the encoded-dot escalation where a crafted `%2e%2e` path is served under a strict narrow gate but canonicalizes _up_ to a broad weak rule — without the guard the broad rule would shallow-merge down the narrow gate's options (e.g. `basicAuth` credentials). (Per the input contract, the `%2e%2e` escalation is **not reachable through h3** — h3 resolves the dots before dispatch, so it serves such a path under the canonical route already. The guard is kept as defense-in-depth for direct/compiled matcher callers and because it is also what makes the `%2f` case safe; the `%2e` cases in `test/merge.test.ts` are synthetic inputs, not h3 traffic, and are labeled as such.) When two patterns are not orderable by containment (`disjoint`/`partial`), the guard fails closed and keeps the served path's rule. Pinned by the two "broader canonical never DOWNGRADES / narrower still OVERRIDES" cases in the "dual-path union" block of `test/merge.test.ts`.
- **`canonicalPath` / `isPathInScope`** (`src/internal/scope.ts`): canonical results are for rule matching and scope checks only — never for routing/dispatch or proxy/redirect forwarding. Proxy/redirect forward `event.url.pathname` **as h3 served it** (unmodified by us; per the input contract only `%2f` is still opaque at that point, so that is what stays opaque to the upstream — `%5c` never survives h3's decode). `isPathInScope` fails closed under **two** interpretations: h3's empty-segment-preserving canonical form AND a separator-run-collapsed form (`SEPARATOR_RUN_RE`, mirroring h3's `%2f`/`%5c` decode). They diverge when a `..` sits next to an empty `//` segment — h3 lets the empty shield the `..` (in scope), but a slash-merging downstream (nginx `merge_slashes`) drops the empty and the `..` traverses out; requiring the collapsed reading in scope too closes that gap (covers literal `//`, encoded `%2f%2f`, and nested variants). `isPathInScope` is scope-only (not used by matching), so this stays strict-fail-closed without affecting dispatch.
- **`%25`-nesting (`%252f`) is deliberate defense-in-depth — keep it, but know what it costs.** h3 never unwraps `%25` and neither does a correctly single-decoding downstream, so treating `%252f` as a separator only matches the reality of a _double-decoding_ downstream (proxy decodes → backend decodes again; the Tomcat/IIS double-decode class). It is **not** purely fail-closed: because the dual-path union may override with an equal-or-more-specific canonical pattern, a `%252f` path can pick up a **narrower, weaker** rule (e.g. a weaker `basicAuth` on a public subtree inside an admin area) that h3 and a single-decoding origin would both serve under the stricter broad rule. A `basicAuth: false` reset cannot cause this (resets are union-only, never strip); only an explicit weaker credential object on a narrower pattern can. It stays regardless because the nesting is baked into h3's `resolveDotSegments` (`ENCODED_SEP_RE = /%(?:25)*(?:2f|5c)/i`) with **no opt-out** — tightening `SEPARATOR_RUN_RE` / `LEADING_SEPARATOR_RUN_RE` alone would desync them from the `canonicalPath` reading they exist to mirror, which is a worse failure than over-decoding. **Tightening is an upstream h3 change (a `decodeSlashes` depth option), not a local one.**
- **Wildcard redirect/proxy scope**: for `base`-scoped rules, out-of-scope canonical traversal must throw 400 — and so must any request whose **raw** path does not literally sit under the base (encoded separator inside the base region, canonical-only match via dot segments): the base cannot be faithfully stripped from the raw path, so the target resolver fails closed rather than forwarding it unstripped (`prepareRuleTarget` in `src/rules/_utils.ts` hoists only static `to`-derived prep; every scope check stays per-request — `resolveRuleTarget` is its one-shot wrapper). Base-less `//` prefixes must be collapsed, never forwarded as protocol-relative URLs.
- **Final-target scope check**: scope is enforced on the **resolved upstream target** (after base strip + `joinURL`), not only the incoming path. A doubled/leading slash or `/./` can make the incoming path canonicalize _inside_ base (an empty `//` segment absorbing a `..`), yet once the base is stripped and the remainder rejoined — `joinURL` collapses the empty segment — the `..%2f` escapes the target's own base the moment a downstream decodes it. The check re-canonicalizes the final target's pathname against the target base (the `to` minus `/**`) and throws 400 on escape. Pinned by the "resolveRuleTarget final-target scope" block and the doubled-slash case in "encoded-separator hardening" (`test/rules.test.ts`).
- **`basicAuth` has `order: -2`** (outer to `headers` at -1 and everything else at 0) so unauthorized requests are never redirected/proxied/cached and never carry `headers` (auth-vs-redirect/proxy pinned in `test/rules.test.ts`, auth-vs-cache in `test/cache.test.ts`).
- **`cors` has `order: -3`** (outer to `basicAuth`) so a CORS **preflight** (`OPTIONS` + `Origin` + `Access-Control-Request-Method`) is answered directly with a 204 + policy headers _before_ the auth gate — browsers send preflights without credentials, so gating them behind `basicAuth` would break CORS, and the preflight response exposes only the CORS policy (no protected data). A non-preflight request still passes through to `basicAuth`; do not reorder `cors` inside auth. Pinned by the "cors rule" preflight/auth-bypass block in `test/rules.test.ts`.
- **Per-request CORS headers never enter cache entries** (`src/cache.ts`, `moveVolatileCorsHeaders`): the ocache glue moves `access-control-allow-origin` / `access-control-allow-credentials` off the response before serialization (re-set on `event.res` so the miss request still sends them; the `cors` rule at order -3 re-appends fresh values on every hit) — otherwise a reflected allowlist origin is baked into the shared cache and served across origins against the entry's own `vary: origin`. Pinned in `test/cache.test.ts`.
- **`credentials: true` + wildcard origin is rejected at normalize time** (`src/normalize.ts`): `origin` unset, `"*"`, or an array containing `"*"` throws — `Access-Control-Allow-Origin: *` is invalid for credentialed requests, and emitting the pair silently misleads. Function origins pass (not statically checkable). Pinned in `test/normalize.test.ts`.
- The regression suites in `test/scope.test.ts`, the "dual-path union" block in `test/merge.test.ts`, and the "encoded-separator hardening" block in `test/rules.test.ts` pin all of the above — do not weaken or delete them.
