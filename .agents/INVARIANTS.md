# Behavioral invariants

- **Merge semantics** (`src/merge.ts`): layers merge least→most specific; object options shallow-merge, non-objects (including `null` — guard the `typeof null === "object"` quirk via `isMergeableObject`) override, `options: false` deletes the accumulated rule (`redirect`/`proxy` accept `false` reset markers like `cache`/`basicAuth`); `route`/`params` take the more specific match, and `params` stays `undefined` when no matched layer carries params (plain/preMerge structural parity). Normalization merges config keys that collide after canonicalization (`"get /x"` vs `"GET /x"`) per rule name instead of dropping the earlier one, and never mutates user config objects.
- **`swr: 0` is valid** (serve stale, revalidate immediately) — never falsy-check swr in normalization.
- **rou3 method shadowing workaround** (`createRulesRouter`): rou3 resolves `methods[m] || methods[""]` per node, so method-agnostic entries are _prepended_ to each method-scoped registration of the same path. Removing this silently drops agnostic rules for method-scoped patterns.
- **Compiler ↔ runtime parity is a tested invariant**: both build the router via `createRulesRouter` and resolve via `createMatcherFromFind`. Any matching/merging change must go through those shared paths (never fork the logic), and `test/compiler.test.ts` must stay green.
- **Cache memoization is instance-scoped** (closure `Map` in the core `createCacheRuleHandler`), keyed `${ruleRoute}:${matchedRoute}` — no `globalThis` state. The core factory **requires** `defineCachedHandler` (h3-rules ships no caching implementation); the ocache-backed handler is the `h3-rules/cache` subpath (`cache` shared instance — module-scoped memoization — plus the ocache-wired `createCacheRuleHandler({ storage?, defaults? })`). `createRouteRulesMatcher` registers **no** default `cache` handler and **throws at construction** when rules use `cache`/`swr` without a `cache` registry entry — an explicit `handlers: { cache: undefined }` (or any own `cache` key) suppresses it; a rule set whose only cache values are `false` resets does not throw (pinned in `test/cache.test.ts`).
- **Match memoization** (`memoizeRouteRulesMatcher`, opt-in by explicitly wrapping a matcher) is exact because a `method + pathname` key fully determines the merged result (params included) — the dual-path/canonical resolution happens _before_ the memo, so it must stay keyed on the **raw** pathname, never the canonical one (pinned by the raw/canonical divergence test in `test/memoize.test.ts` — canonical keying is an auth bypass). Memoized `MatchResult`s are shared across requests (treat as immutable). FIFO-capped (default 1024, asserted). ~8× on warm paths; the biggest per-request win. Memoization is **decoupled** from matcher construction: neither `createRouteRulesMatcher` (runtime) nor `createMatcherFromFind` (compiled integration point) wires it — keeping the reference out of both lets an un-memoized bundle tree-shake `memoizeRouteRulesMatcher` away (~240 B minified). All consumers opt in by composing `memoizeRouteRulesMatcher(createRouteRulesMatcher(...))` or `memoizeRouteRulesMatcher(createMatcherFromFind(find))`; do not re-add a `memoize` option to either constructor (pinned by `test/treeshake.test.ts`). The compiler's optional `matcher` export (`compileRouteRules(..., { matcher })`) can emit that composition in **generated source** — but only imports `memoizeRouteRulesMatcher` when `matcher.memoize` is set, so an un-memoized matcher export still tree-shakes it away; this composes the runtime helper, it does not move memoization into the compiled integration point.
- **Chain pre-merge** (`preMerge` option, `src/internal/premerge.ts`) resolves each pattern's subsumption chain at build/startup so per-request resolution takes only the **most specific matched layer** (~20% cold-path win; rou3 lookup is the remaining floor). Exactness rests on three pillars — break any and it is silently wrong:
  1. **Chain-cleanness**: every overlapping pattern pair must be strictly ordered by containment — verdicts come from rou3 `compareRoutes` (`"equal"` and `"partial"` — e.g. `/a/*/c` vs `/a/b/*`, or undecidable regex pairs — `preMergeRuleLayers` must keep **throwing** on them). The **runtime** matcher surfaces that throw (a misconfigured `preMerge` is a startup error). The **compiler** is fail-safe: `resolveEffectivePreMerge` (`src/compiler/compile.ts`) catches the throw, `console.warn`s, and falls back to plain compilation — resolved identically across `compileFindRouteRules`/`compileHandlersImport`/`compileRouteRules` (warned once) so imports never desync from references. Do not re-implement pattern semantics locally; `test/premerge.test.ts` pins the `compareRoutes` verdicts preMerge relies on.
  2. **Method matrix**: a `(M, path)` registration is materialized whenever any chain member has `M`-scoped rules, else a method-scoped broad rule would be dropped by take-last (pinned by the "method matrix" test).
  3. **Per-rule params**: `paramRoutes` records exactly the chain patterns that contributed to a rule; runtime merges params from exactly those matched layers — never just the last layer's params.
- preMerge/memoize parity with the plain matcher is pinned over probe grids in `test/premerge.test.ts` (plain, compiled, and preMerge composed with `memoizeRouteRulesMatcher`) and `test/memoize.test.ts`; compiled/runtime method-fallback parity (rou3#190 regression) in `test/compiler.test.ts`. The grid fixture/probes/snapshot live in `test/_fixture.ts` (shared so compiler and premerge grids cannot drift; the snapshot includes `method` — dropping fields weakens the differential). Any matching/merging change must keep all grids green.

# Contracts to keep in sync

- Adding a rule handler: add a module under `src/rules/`, register it in `ruleHandlers` (`src/rules/index.ts`), `RUNTIME_RULE_NAMES` (`src/compiler/runtime-rules.ts`), **and** export it by name from `src/index.ts` — tests assert registry ↔ names ↔ named exports all match. Exception: the **opt-in subpath handlers** `cache` and `proxy` are in `RUNTIME_RULE_NAMES` (and `SUBPATH_RULE_SOURCES`) but **not** in `ruleHandlers` / `src/index.ts` — their handlers are subpath exports (`cache`→`src/cache.ts`/`h3-rules/cache`, the only module allowed to import ocache; `proxy`→`src/proxy.ts`/`h3-rules/proxy`, the only module allowed to import h3's `proxyRequest`). `redirect` (core) and `proxy` (subpath) share wildcard-target resolution (incl. the scope check) via `resolveRuleTarget` (`src/rules/_utils.ts`, ufo) — that shared module keeps ufo in the core surface via `redirect`, but `proxyRequest` stays out of it.
- Compiled-mode import contract: codegen imports **exactly the handlers the rule set uses**, as named exports bound `<ns>$<name>` (`compileHandlersImport(rules)` emits `import { headers as __ruleHandlers__$headers, … }`; empty rule sets emit no import). The binding local (`<ns>$<name>`) is invariant, keyed on the **rule name** — but each rule's import `source` and export name come from its `runtimeRules` entry (`Record<name, string | {source, export?}>`; `resolveRuntimeRule` reads a bare string as the source, `export`→rule name). The caller's `runtimeRules` is merged **over** `DEFAULT_RUNTIME_RULES` (= every built-in → `"h3-rules"`, except the `SUBPATH_RULE_SOURCES` entries `cache` → `"h3-rules/cache"` and `proxy` → `"h3-rules/proxy"`) by `resolveRuntimeRules`, so callers list only additions/overrides and the built-ins stay registered. There is **no** ambient default source — a custom rule's entry must name its module (no more `handlersImportId`). Handlers are grouped by resolved source into one import statement each (sources sorted; specifiers in binding order); `usedRuleHandlerNames`/serialize gate on record membership (`isRuntimeRule`), not the old name array. This is what keeps unused handlers + their deps (ocache/ufo) tree-shakeable in compiled bundles (`pnpm bench:size` measures it). Each entry's `source` module must export the handler under its resolved export name (`h3-rules` exports the built-ins, `h3-rules/cache` exports `cache` — a shared instance with module-scoped memoization; runtime matchers may register a per-instance one instead). Changing the **binding** scheme (not the source/export resolution) breaks every consumer's codegen.
- `RouteRuleConfig` is a **closed interface** (no index signature — typos are compile errors); `RouteRules` stays **open** (`[key: string]: unknown`) so the runtime can carry arbitrary rule names on matched results. Custom / data-only keys are declared via module augmentation (README "Extending rule types"); do not add `isr`/`prerender`/`static` or other consumer-specific keys to the core types. Regardless of typing, unknown keys must keep flowing through normalize/match/merge as data-only rules (no handler assumed) — closing `RouteRuleConfig` is a **type-level** change only. Pinned by `test/types.test-d.ts` (typecheck-only; the test/bench fixtures' ad-hoc keys are re-declared in `test/_augment.ts`).
- **Compiler auto-normalizes; the runtime matcher does not.** Every `h3-rules/compiler` entrypoint (`compileFindRouteRules`/`compileHandlersImport`/`compileRouteRules`) calls `normalizeRouteRules` on its input (build-time, so the pass is free; raw config would otherwise silently mis-compile — an unexpanded `swr` shortcut compiles as a data-only rule with no cache handler import). This makes **`normalizeRouteRules` idempotency a contract** (pinned in `test/normalize.test.ts`): consumers may pass already-normalized rule sets, so every normalization step must re-apply as a no-op — a new step that cannot must trigger a redesign of the compiler input handling, not deletion of the test. `createRouteRulesMatcher` intentionally keeps taking **normalized** rules (normalization stays out of runtime bundles; compiled-path parity — baked-in rules have no runtime normalization point); `routeRules()` is the auto-normalizing runtime entry point.
- Normalized `redirect`/`proxy` options carry a first-class `base` field (rule key minus `/**`; `createRulesRouter` prefixes the matcher `baseURL` onto it at registration, since the scope check runs against the full request path) — public API, not an internal flag; the compiler serializes it (post-prefix).
- Compiled rule options must survive a JSON round-trip: the compiler **throws** on functions, nested `undefined`, and class instances (`Date`, `RegExp`, …) instead of letting `JSON.stringify` silently diverge from the runtime matcher, and validates that handler binding names (`runtimeRules` keys, per-rule `export` names, `handlersImportName`) are valid JS identifiers.
