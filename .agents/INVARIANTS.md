# Behavioral invariants

- **Merge semantics** (`src/merge.ts`): layers merge least→most specific; object options shallow-merge, non-objects (including `null` — guard the `typeof null === "object"` quirk via `isMergeableObject`) override, `options: false` deletes the accumulated rule (`redirect`/`proxy` accept `false` reset markers like `cache`/`basicAuth`); `route`/`params` take the more specific match, and `params` stays `undefined` when no matched layer carries params (plain/preMerge structural parity). Normalization merges config keys that collide after canonicalization (`"get /x"` vs `"GET /x"`) per rule name instead of dropping the earlier one, and never mutates user config objects.
- **`swr: 0` is valid** (serve stale, revalidate immediately) — never falsy-check swr in normalization.
- **rou3 method shadowing workaround** (`createRulesRouter`): rou3 resolves `methods[m] || methods[""]` per node, so method-agnostic entries are _prepended_ to each method-scoped registration of the same path. Removing this silently drops agnostic rules for method-scoped patterns.
- **Compiler ↔ runtime parity is a tested invariant**: both build the router via `createRulesRouter` and resolve via `createMatcherFromFind`. Any matching/merging change must go through those shared paths (never fork the logic), and `test/compiler.test.ts` must stay green.
- **Cache memoization is instance-scoped** (closure `Map` in `createCacheRuleHandler`), keyed `${ruleRoute}:${matchedRoute}` — no `globalThis` state.
- **Match memoization** (`memoizeRouteRulesMatcher`, opt-in via the `memoize` matcher option) is exact because a `method + pathname` key fully determines the merged result (params included) — the dual-path/canonical resolution happens _before_ the memo, so it must stay keyed on the **raw** pathname, never the canonical one (pinned by the raw/canonical divergence test in `test/memoize.test.ts` — canonical keying is an auth bypass). Memoized `MatchResult`s are shared across requests (treat as immutable). FIFO-capped (default 1024, asserted). ~8× on warm paths; the biggest per-request win.
- **Chain pre-merge** (`preMerge` option, `src/internal/premerge.ts`) resolves each pattern's subsumption chain at build/startup so per-request resolution takes only the **most specific matched layer** (~20% cold-path win; rou3 lookup is the remaining floor). Exactness rests on three pillars — break any and it is silently wrong:
  1. **Chain-cleanness**: every overlapping pattern pair must be strictly ordered by containment — verdicts come from rou3 `compareRoutes` (`"equal"` and `"partial"` — e.g. `/a/*/c` vs `/a/b/*`, or undecidable regex pairs — must keep **throwing** at build time). Do not re-implement pattern semantics locally; `test/premerge.test.ts` pins the `compareRoutes` verdicts preMerge relies on.
  2. **Method matrix**: a `(M, path)` registration is materialized whenever any chain member has `M`-scoped rules, else a method-scoped broad rule would be dropped by take-last (pinned by the "method matrix" test).
  3. **Per-rule params**: `paramRoutes` records exactly the chain patterns that contributed to a rule; runtime merges params from exactly those matched layers — never just the last layer's params.
- preMerge/memoize parity with the plain matcher is pinned over probe grids in `test/premerge.test.ts` (plain, compiled, and composed `preMerge+memoize`) and `test/memoize.test.ts`; compiled/runtime method-fallback parity (rou3#190 regression) in `test/compiler.test.ts`. The grid fixture/probes/snapshot live in `test/_fixture.ts` (shared so compiler and premerge grids cannot drift; the snapshot includes `method` — dropping fields weakens the differential). Any matching/merging change must keep all grids green.

# Contracts to keep in sync

- Adding a rule handler: add a module under `src/rules/`, register it in `ruleHandlers` (`src/rules/index.ts`), `RUNTIME_RULE_NAMES` (`src/compiler.ts`), **and** export it by name from `src/index.ts` — tests assert registry ↔ names ↔ named exports all match. `redirect`/`proxy` share wildcard-target resolution (incl. the scope check) via `resolveRuleTarget` (`src/rules/_utils.ts`).
- Compiled-mode import contract: codegen imports **exactly the handlers the rule set uses**, as named exports bound `<ns>$<name>` (`compileHandlersImport(rules)` emits `import { headers as __ruleHandlers__$headers, … }`; empty rule sets emit no import). This is what keeps unused handlers + their deps (ocache/ufo) tree-shakeable in compiled bundles (`pnpm bench:size` measures it). The handlers module (`handlersImportId`) must export each runtime rule name; `cache` is a shared default instance (`src/rules/cache.ts`) — module-scoped memoization, replaced per-instance by runtime matchers. Changing the binding scheme breaks every consumer's codegen.
- `RouteRuleConfig` / `RouteRules` are **open interfaces** for module augmentation (README "Extending rule types"). Do not add `isr`/`prerender`/`static` or other consumer-specific keys to core types; unknown keys must keep flowing through normalize/match/merge as data-only rules (no handler assumed).
- Normalized `redirect`/`proxy` options carry a first-class `base` field (rule key minus `/**`; `createRulesRouter` prefixes the matcher `baseURL` onto it at registration, since the scope check runs against the full request path) — public API, not an internal flag; the compiler serializes it (post-prefix).
- Compiled rule options must survive a JSON round-trip: the compiler **throws** on functions, nested `undefined`, and class instances (`Date`, `RegExp`, …) instead of letting `JSON.stringify` silently diverge from the runtime matcher, and validates that handler binding names (`runtimeRules`, `handlersImportName`) are valid JS identifiers.
